# Codexでキャラクター設定を追加する運用

このプロジェクトでは、WebフォームではなくCodexへの依頼を主な入力口にする。

## ユーザーからCodexへの依頼例

```text
この情報を demo-character の公式設定候補として取り込んで。
確定できる内容は character.json に反映し、未確定や矛盾しそうな内容は inbox に残して。

（ここに原典、メモ、URL、会話ログ、画像生成結果の評価などを貼る）
```

```text
demo-character の好きな食べ物を「いちご大福」に確定して。
公式サイトとAI向けプロンプトにも反映して。
```

```text
この2つの設定が矛盾していないか確認して。
矛盾していたら、どちらを公式にするべきか判断材料をまとめて。
```

## Codex側の処理方針

最重要ルール: 足りない情報は推測で補完しない。

- 未提供の事実は `未定義` として扱う
- 好きな食べ物、苦手なもの、一人称、家族構成、出身地などを勝手に作らない
- 苗字・名前の形式が明示されていないキャラクター名は、無理に分解しない
- 公式化した内容と未定義の内容を分けて報告する

1. 原典やメモを読む
2. 既存の `character.json` と照合する
3. 内容を分類する

- 確定: 公式設定として `character.json` に反映
- 未確定: `content/inbox/{characterId}/` にメモとして保存
- 矛盾候補: 既存設定との差分を明記して確認待ちにする
- 派生アイデア: 公式化せず、候補として残す

4. `npm.cmd run build` で生成結果を確認する
5. 変更点を簡潔に報告する

## 文字化け対策

日本語を含む正本ファイルはUTF-8で扱う。PowerShellやターミナルの表示だけが文字化けすることがあるため、表示が崩れたテキストをそのまま正本へ戻してはいけない。

- `Get-Content` の表示が典型的なmojibake文字列として崩れた場合は、まずNode.jsでUTF-8読みを確認する。
- ユーザー提供文、Git上の差分、ブラウザ表示を優先し、文字化けしたターミナル出力を根拠にしない。
- 日本語本文を編集したら `npm.cmd run check` を実行する。
- 詳細は `docs/encoding-safety.md` を参照する。

## ローカルdevサーバー

ページを目視確認するときは `npm.cmd run dev` を使う。

- `PORT` 未指定時は、ワークツリーのパスから安定した既定ポートを自動選択する。
- 複数ワークツリーを同時に開いている場合でも、それぞれ別ポートになりやすい。
- 起動ログに出る `Local: http://127.0.0.1:{port}/` が正しい確認URL。
- 既にそのポートが使われている場合は、近い空きポートへフォールバックする。
- 特定ポートで開きたい場合だけ、PowerShellで `$env:PORT=4180; npm.cmd run dev` のように明示する。

## Git / Pull Request 運用

協力者から編集を受け取ることを前提に、通常はPull Request運用をデフォルトにする。

1. `main` から作業ブランチを切る
2. 公式設定やドキュメントを更新する
3. `npm.cmd run check` を実行し、必要に応じて `npm.cmd run build` とローカル表示確認を行う
4. 変更をコミットする
5. ブランチをpushし、Pull Requestを作る
6. レビュー後に `main` へmergeする

### 既存PRに追加変更を載せるとき

追加変更をpushする前に、必ず対象PRの状態を確認する。

- `open` のPRだけが追加pushの対象になる。
- `merged` / `closed` のPRは更新できない。新しい作業ブランチを `origin/main` から切り、新しいPRを作る。
- マージ済みPRのheadブランチへpushしただけでは、新しい変更はmainにも既存PRにも入らない。
- その状態で「PRにpush済み」と報告しない。必ず新しいPR番号を作ってから報告する。
- ブランチを使い回している場合も、PRがマージされたら次の変更では新ブランチを使うのを原則にする。

`main` への直接pushは、ユーザーが `ship-main` のように明示した場合、または自己利用の小さな変更で直pushが適切な場合だけにする。

## 反映時の原則

- 公式設定にないことは勝手に断定しない
- 足りない設定は推測で補完せず、`未定義` として残す
- 出典がある場合は `sources` に残す
- 迷う内容は `draft` として設定する
- 年齢、身長体重、口調など、AI出力に強く影響する項目は優先して明確化する
- 画像・動画生成に関わる外見、服装、色、持ち物、動きは `promptGuidance` にも反映する
