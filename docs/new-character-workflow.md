# 新規キャラクターページ作成ワークフロー

このドキュメントは、新しいキャラクターページを Codex がユーザと対話しながら作るための手順書です。

## 最重要ルール

- 足りない情報を推測で補完しない。
- 未確定の情報は `未定義` として扱う。
- 任意項目に情報がない場合は、空配列または項目自体を省略してよい。
- キャラクター名を苗字・名前に分割しない。ユーザが明示した場合だけ分割情報を書く。
- 見た目、年齢、性格、関係性、世界観、時系列は、ユーザが出した情報だけを公式設定として扱う。
- 迷った情報は `content/inbox/{characterId}/` にメモとして残し、公式設定には入れない。

## 最初に聞くこと

ユーザに一度に全部聞かない。まず以下だけ確認する。

1. キャラクター表示名
2. URL用ID
3. 1〜3文の概要
4. 既に確定しているプロフィール項目
5. イメージカラー

IDは小文字英数字とハイフンを推奨する。例: `zannenin`, `my-character`

ユーザが大文字、アンダースコア、空白などを含むIDを指定した場合は、ビルド検証に通る英小文字・数字・ハイフン形式へ正規化する。元の指定IDは `content/inbox/{characterId}/README.md` に確認事項として残す。

## 対話の進め方

まずユーザから出た情報をそのまま整理する。

```text
確認できた公式設定:
- ...

未定義のままにする項目:
- ...

追加で聞くとよさそうな項目:
- ...
```

追加質問は最大3つまでにする。ユーザが「まず入れられるところから」と言った場合は、質問を増やさず初期ページを作る。

## 初期作成後の次アクション提案

初めてキャラクターを作った直後は、完了報告だけで終わらず、次に決めるとよい項目を提案する。

提案は、現在の `character.json` を見て未定義や空配列が多い領域から選ぶ。公式設定として勝手に補完せず、「次に聞くとよいこと」として扱う。

優先順の目安:

1. イメージカラー: サイト全体の印象がすぐ変わり、キャラクター固有の雰囲気を決めやすい。`theme.primary`, `theme.secondary`, `theme.accent`, `theme.paper` を中心に反映する。
2. 概要・キャッチフレーズ: トップ表示、OGP、AI向けプロンプトの芯になる。
3. 基本プロフィール: 年齢、身長、好きな食べ物、一人称など、短く確定しやすい項目を埋める。
4. 外見・服装・持ち物: 画像生成AIへの影響が大きいため、確定している要素だけを `settings` と `promptGuidance.image` に反映する。
5. 話し方・性格・世界観: 会話AIやText-to-Text出力の品質に直結するため、断定できる内容だけを整理する。
6. ヒーローバナー・OGP: ページの第一印象とSNS共有時の見え方に直結する。画像がある場合は `brandAssets.banner` を設定し、生成された `ogp.png` を確認する。
7. 二次創作ガイドライン: キャラクターの扱い方、許可範囲、禁止事項を早めに整理すると、公開後の使われ方を案内しやすい。
8. リンク、ビジュアル資料、年表: 素材や運用方針が出てきた段階で追加する。

特に初期情報が名前とIDだけの場合は、まずイメージカラーから聞くのを推奨する。質問例:

```text
次はイメージカラーから決めるのがおすすめです。ユズの色として、以下を教えてください。

1. メインカラー:
2. サブカラー:
3. アクセントカラー:

色名だけでも、カラーコードでも大丈夫です。迷っている色は候補として inbox に残します。
```

次アクションの提案では、質問を最大3つまでにする。ユーザが「おすすめで進めて」と言った場合でも、未提供の事実は公式化せず、色の候補や方向性は `draft` または inbox の確認待ちとして扱う。

## 作成するファイル

新規キャラクターIDを `{characterId}` とする。

```text
content/
  characters/
    {characterId}/
      character.json
      assets/
        brand/
        links/
  inbox/
    {characterId}/
      README.md
```

画像がまだない場合、`assets/` は空でもよい。

  "theme": {
    "primary": "#151217",
    "secondary": "#d4a72c",
    "accent": "#fff3c4",
    "paper": "#fffaf0",
    "panel": "#ffffff",
    "text": "#1f1a12",
    "muted": "#756b5e"
  },
  "profile": {
    "呼称": "{キャラクター名}",
    "年齢": "未定義",
    "身長": "未定義",
    "体重": "未定義",
    "好きな食べ物": "未定義",

## 情報の入れ方

### profile

年齢、身長、体重、出身、好きな食べ物、一人称、愛称など、短い項目を入れる。

不明な値は `未定義`。

### glossary

用語、組織、地名、サービス名、固有名詞を入れる。

```json
{
  "term": "用語名",
  "definition": "説明"
}
```

### settings

世界観、性格、話し方、衣装、能力、関係性など、本文として読ませたい設定を入れる。

```json
{
  "title": "設定名",
  "body": "設定本文",
  "status": "official"
}
```

未確定の場合は `status` を `draft` にする。

### sideFlavors

本筋ではないが面白いサブ設定、ネタ、周辺文脈を入れる。

### timeline

日付がある出来事を入れる。年齢固定キャラクターの場合は、作中時間と現実の活動履歴を混ぜない。

- `timelineType: "fictional"`: 架空年表。キャラクター内時間、作中前史、自己申告の誕生や成長など。
- `timelineType: "real"`: 実年表。公開日、SNSでの出来事、活動シーズン、サービス終了など。

```json
{
  "date": "2026-04-29",
  "event": "出来事",
  "detail": "補足",
  "timelineType": "real"
}
```

期間は `2026-04-27〜2026-04-28` のように書く。

年齢が固定の場合、`0歳頃`, `12歳頃` のような出来事は `fictional` に入れる。実時間の経過で現在年齢を増やさない。

### links / contentLinks

SNSは `links`、公式ゲーム・Discord・資料集・ショップなどは `contentLinks` に入れる。

```json
{
  "label": "YouTube",
  "url": "https://example.com",
  "type": "social",
  "displayUrl": "YouTube"
}
```

### visualReferences

画像がある場合だけ入れる。

```json
{
  "label": "三面図",
  "path": "assets/outfit-reference.png",
  "type": "outfit",
  "description": "通常衣装の三面図。"
}
```

画像ファイルは `content/characters/{characterId}/assets/` 以下に置く。

### brandAssets / OGP

ヒーローバナーやロゴ画像がある場合だけ入れる。ページ上部、ルートページのキャラクターカード、OGPに強く影響するため、初期ページができた後の次アクションとして提案してよい。

```json
{
  "brandAssets": {
    "banner": {
      "source": "assets/brand/banner-source.png",
      "alt": "キャラクターのヒーローバナー"
    },
    "logo": {
      "source": "assets/brand/logo-source.png",
      "alt": "キャラクターロゴ"
    }
  }
}
```

画像ファイルは `content/characters/{characterId}/assets/brand/` 以下に置く。バナーだけ、ロゴだけでもよい。

バナー画像を追加したら、`npm.cmd run build` 後に以下を確認する。

- `dist/{characterId}/assets/generated/brand/banner-logo.webp`
- `dist/{characterId}/assets/generated/ogp.png`
- ローカルページのヒーローバナー表示
- ルートページ `dist/index.html` のキャラクターカードにバナーが表示されること
- OGP画像内の日本語テキストが文字化けしていないこと

バナーやロゴがない場合でも、ビルドスクリプトが `ogp.png` を生成する。日本語名が入るキャラクターでは、生成後のOGPを目視確認する。

### AI向けプロンプト誘導メタデータ

キャラクターページのトップURLがクロールされた場合に、生成AIが各種Markdownプロンプトへ辿りやすいよう、ビルドスクリプトはトップページの `<head>` に次の情報を出力する。

- `link rel="alternate" type="text/markdown"` による各プロンプトMDへのリンク
- `meta name="ai:prompt:*"` によるプロンプトMD URL
- `script type="application/json" id="ai-prompt-manifest"` によるプロンプト一覧
- JSON-LD `subjectOf` によるSchema.org `CreativeWork` 参照

新規キャラクター作成後は、トップページHTMLに `ai-prompt-manifest` と `../prompts/{characterId}/` への導線が出ているか確認する。Markdown本文そのものをHTMLメタデータへ丸ごと埋め込むとページが重くなり、検索スニペットやSNSクローラの挙動も不安定になりやすいため、原則はMarkdown URLを明示する方式を優先する。

## 二次創作ガイドライン

ユーザが希望した場合だけ `fanworkGuidelines` を追加する。希望未確認のまま勝手に公式ルール化しない。

緩い版・厳しい版の明確なテンプレートは `docs/fanwork-guideline-templates.md` を参照する。

初回キャラクター作成後の次アクションとして、プロフィール、外見、話し方のどれかが固まり始めたら、二次創作ガイドラインを提案してよい。提案時は最大3つだけ確認する。

権利者情報はメインページと二次創作ガイドラインの両方に表示される。権利者名や管理者が未確認の場合は推測せず、`rights` に `未定義` として置く。確定後に更新する。

ガイドラインを公開する場合は、末尾の `revisionHistory` も必ず入れる。初回は作成日または公開日と、初版公開・主要な変更点を短く残す。ページ上ではPDF出力ボタンからブラウザの印刷/PDF保存を使えることも確認する。

```text
二次創作ガイドラインも初期案を作れます。まず以下だけ確認します。

1. 二次創作・AI生成・ファンアートは許可寄りでよいか:
2. 商用利用は許可、応相談、禁止のどれにするか:
3. 絶対に避けたい表現や使い方:
```

ユーザが「おすすめで進めて」と言った場合は、寛容寄りの初期案を `draft` ではなく `official` として置くか、確認待ちの初期案として inbox に残すかを文脈で判断する。迷う場合は `content/inbox/{characterId}/` に確認待ちとして残す。

寛容寄りの初期案が必要な場合は、次の方針でスキャフォールドする。

- 二次創作、AI生成、ファンアート、動画、配信、記事化を許可寄りにする。
- 商用利用は応相談。
- メディアサイト、動画投稿サイト、SNSなどのPVや再生数に応じた広告収入は許可。
- 公式と誤認させる表示は禁止。
- キャラクターや第三者を傷つける使い方は禁止。

追加する場合の最小形:

```json
{
  "fanworkGuidelines": {
    "title": "{キャラクター名} 二次創作ガイドライン",
    "summary": "{キャラクター名} の二次創作、AI生成、ファンアート、動画、配信、記事化に関する初期ガイドラインです。",
    "status": "official",
    "contact": "商用利用や判断に迷う利用は、公式窓口または作者へ相談してください。",
    "revisionHistory": [
      {
        "date": "YYYY-MM-DD",
        "summary": "初版公開。"
      }
    ],
    "sections": [
      {
        "title": "許可していること",
        "items": [
          "二次創作、AI生成、ファンアート、動画、配信、記事化は許可します。",
          "SNS、動画投稿サイト、メディアサイトなどでの紹介や投稿を許可します。",
          "投稿先の仕様による広告収入や収益化は許可します。"
        ]
      },
      {
        "title": "相談が必要なこと",
        "items": [
          "グッズ販売、商品化、有料素材化などの商用利用は事前相談とします。",
          "公式企画や公式素材と誤認されやすい利用は事前相談とします。"
        ]
      },
      {
        "title": "禁止していること",
        "items": [
          "公式と誤認させる表示は禁止します。",
          "キャラクターや第三者を傷つける使い方は禁止します。",
          "公式設定と矛盾する内容を公式設定として広めることは禁止します。"
        ]
      }
    ]
  }
}
```

キャラクター固有の禁止事項や、年齢・外見・世界観に関わる注意点がある場合は、ユーザが提供した範囲だけを反映する。足りない禁止事項を推測で足さない。

## 作業手順

1. `content/characters/{characterId}/character.json` を作る。
2. `content/inbox/{characterId}/README.md` を作り、未整理情報や今後確認したいことを書く。
3. 必要なら画像を `assets/` に置く。ヒーローバナーやロゴは `assets/brand/`、三面図などの資料は `assets/` 以下に置く。
4. ヒーローバナーやロゴがある場合は `brandAssets` を追加する。
5. ユーザが希望した場合は `fanworkGuidelines` を追加する。希望未確認の場合は、次に確認したい項目として inbox または完了報告に残す。
6. `npm.cmd run build` を実行する。
7. `npm.cmd run check` を実行する。
8. ローカルで `http://127.0.0.1:4173/{characterId}/` を確認する。
9. `dist/{characterId}/assets/generated/ogp.png` を目視確認する。バナーやロゴを入れた場合は、ヒーローバナー表示、ルートページのキャラクターカード、OGPも確認する。
10. トップページHTMLにAI向けプロンプト誘導メタデータが出ているか確認する。
11. 変更内容、未定義のままにした項目、次に聞くとよい項目をユーザに報告する。
12. 初回作成時は、次アクションとしてイメージカラー、概要・キャッチフレーズ、基本プロフィール、ヒーローバナー・OGP、二次創作ガイドラインなどから最大3つを提案する。

## ユーザへの確認テンプレート

```text
まず初期ページを作るために、以下だけ確認します。

1. キャラクター名:
2. URL用ID:
3. 1〜3文の概要:
4. 確定しているプロフィール:
5. イメージカラー:

未確定の項目は推測せず、未定義のまま進めます。
```

## 完了報告テンプレート

```text
初期キャラクターページを作成しました。

作成:
- content/characters/{characterId}/character.json
- content/inbox/{characterId}/README.md

反映した公式設定:
- ...

未定義のままにした項目:
- ...

確認:
- npm.cmd run build
- npm.cmd run check

次におすすめ:
- まずイメージカラーを決めると、公式サイトの印象を作りやすいです。
- 次に、概要・キャッチフレーズ・基本プロフィールのどれを埋めるか選べます。
- ヒーローバナーやOGP画像を入れると、ページ上部とSNS共有時の見栄えが良くなります。
- 公開を意識する場合は、二次創作ガイドラインの初期案も作れます。

次に聞く質問:
1. メインカラー:
2. サブカラー:
3. アクセントカラー:
```
